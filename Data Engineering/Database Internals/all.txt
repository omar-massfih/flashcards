How does a storage engine differ from a database management system?	A storage engine handles low-level data storage and retrieval, while a DBMS adds higher-level features such as schema, queries, indexes, and transactions.
Why can storage engines treat keys and values as arbitrary byte sequences?	Their meaning, sorting, and representation semantics are defined by higher-level database components rather than by the storage engine itself.
Why are pluggable storage engines useful in database systems?	They let database developers reuse existing engines and switch to engines that better fit particular workloads.
What is the best way to compare databases for a real application?	Simulate the intended workloads in a production-like environment and measure the performance and operational metrics that matter to you.
Why are long-running database tests important during evaluation?	Some performance, scalability, and operational problems appear only over time or as capacity grows.
Why can operational friendliness matter more than raw performance when choosing a database?	A database that is easier to run, debug, and recover is often more valuable than one that is faster but less reliable.
Why should database evaluation account for schema, record sizes, access patterns, and read/write rates?	Those variables determine whether the database can support the required queries, scale appropriately, and be operated safely over time.
Why can reading database source code be useful even if you are only operating the database?	It helps you interpret logs, understand configuration, debug failures, and reduce recovery time when problems occur.
Why should generic benchmarks such as YCSB be used cautiously?	Generic workloads can produce misleading conclusions unless they are tailored to real production conditions and access patterns.
Why are benchmarks useful beyond comparing database vendors?	They also help define service expectations, understand system requirements, and support capacity planning.
Why should teams test new database versions before rolling them out?	New releases can improve performance and security, but they can also introduce regressions, new bugs, or unexpected behavior.
Why is designing a storage engine more than implementing a textbook data structure?	It also requires decisions about physical layout, serialization, garbage collection, concurrency, crash safety, and integration with database semantics.
Why do storage-engine design decisions inevitably involve trade-offs?	Optimizing for one goal, such as write speed, read order, density, or simplicity, usually makes another goal harder to achieve.
Why is there no universally optimal storage engine?	Different workloads and operational priorities require different trade-offs, so no single engine is best for every use case.
Why do different storage engines end up with very different designs?	They optimize for different goals, such as read latency, write latency, storage density, or operational simplicity.
What is the job of the query optimizer in a DBMS?	It removes impossible or redundant query parts and chooses the most efficient execution plan based on statistics and data placement.
What is concurrency control in DBMS architecture?	Coordinating concurrent operations so the database preserves logical and physical integrity while still executing efficiently.
Why are main-memory DBMSs attractive?	Memory access is far faster than disk access, and programming against memory is simpler than programming against disk-oriented storage.
Why are RAM cost and volatility major limits for main-memory DBMSs?	RAM is still expensive relative to persistent storage, and its contents can be lost on crashes or power failures without extra durability measures.
How do memory-based DBMSs usually provide durability?	They write changes to a sequential log and maintain a disk-resident backup for recovery.
What is checkpointing in an in-memory DBMS?	Applying batches of log records to a backup snapshot so older log segments can be discarded and recovery can start from a newer point.
Why is an in-memory DBMS not just a disk-based DBMS with a huge cache?	In-memory systems can use different layouts, data structures, and optimizations than disk-oriented systems built around page and serialization overhead.
When is a row-oriented data layout a good fit?	When workloads usually read or write most fields of a record together, such as point queries and range scans over whole rows.
Why does row-oriented layout improve record access locality?	Values belonging to the same row are stored together, so reading one record tends to fetch its related fields in the same disk block.
When is a column-oriented data layout a good fit?	When workloads scan many rows but touch only a subset of columns, especially for analytics and aggregates.
Why can column-oriented layout outperform row-oriented layout for analytics?	It reads only needed columns and benefits from better cache utilization, compression, and vectorized processing.
Why should column-oriented stores not be confused with wide-column stores?	Wide-column stores group columns into families but still organize data for key-based access to rows within those families rather than pure analytical column scans.
What is a heap file in database storage?	A data file where records are not stored in any required key order and are typically placed in write order.
What is an index-organized table in database storage?	A table that stores the actual data records inside the index structure in key order.
How do primary and secondary indexes differ in database storage?	A primary index organizes access to the main data records, while secondary indexes provide alternative lookup paths using other fields.
What is the difference between clustered and nonclustered indexes?	A clustered index keeps data records in search-key order, while a nonclustered index points to records stored in a different order.
Why might a storage engine use the primary index as an indirection layer for secondary indexes?	It reduces the cost of updating secondary pointers when records move, at the expense of an extra lookup on reads.
What three variables distinguish many storage-engine designs?	Buffering, mutability versus immutability, and ordered versus unordered storage.
Why does ordering matter in storage-engine design?	Ordered storage supports efficient range scans, while unordered or insertion-order storage can simplify and speed writes.
Why are binary search trees a poor fit for disk-based storage?	Their low fanout creates tall trees with many pointer traversals and frequent rebalancing, which leads to too many costly disk accesses and pointer updates.
Why does a disk-friendly tree structure need high fanout and low height?	Higher fanout stores more keys per node and reduces the number of levels and disk seeks needed during lookup.
What is fanout in a tree-based storage structure?	The maximum number of child pointers a node can have.
How are fanout and tree height related?	Higher fanout lowers tree height because each node covers a larger part of the key space.
Why are B-trees well suited for disk-based storage?	They keep many keys in each node, reduce tree height, improve locality, and minimize the number of disk seeks per lookup.
Why can B-trees support both point queries and range scans efficiently?	Keys are stored in sorted order, so the tree can both locate exact items quickly and iterate through key ranges in order.
What is a B+-tree?	A B-tree variant that stores values only in leaf nodes while internal nodes store only separator keys and child pointers.
Why are B+-trees commonly treated as the default form of B-trees?	They are the widespread practical variant used by many storage engines and textbooks often refer to them simply as B-trees.
Why can leaf-level sibling pointers be useful in B-trees?	They let range scans continue across neighboring leaves without repeatedly going back up to parent nodes.
Why does B-tree lookup require relatively few disk seeks even for very large datasets?	Each node contains many keys, so one disk read narrows the search space by a large factor.
What causes a B-tree node split?	An insertion overflows a node beyond its maximum capacity.
What happens during a B-tree split?	A new sibling node is allocated, roughly half the entries move to it, and a separator key is promoted to the parent.
Why can a split propagate all the way to the root in a B-tree?	If the parent is already full, promoting the separator overflows that parent too, forcing recursive splits upward.
What happens when a B-tree root splits?	A new root is created above the old root and its new sibling, increasing the tree height by one.
What causes a B-tree node merge?	A deletion leaves neighboring nodes underfull enough that their contents fit into a single node.
What happens during a B-tree merge?	Entries from one sibling are copied into the other, the parent's separator or pointer is removed, and one node disappears.
Why can merges propagate upward in a B-tree?	Removing a separator or pointer from a parent can make that parent underfull as well.
Why can B-trees tolerate node occupancy well below 100%?	Keeping slack space reduces the frequency of splits and merges while still preserving strong lookup performance.
Why can sibling links be useful in a B-tree?	They let the engine move to neighboring nodes on the same level without first climbing back up through parent nodes.
What trade-off comes with storing sibling links in a B-tree?	Splits and merges become more complex because neighboring pages' sibling pointers must also be updated.
Why does a B-tree internal node have one more child pointer than separator keys?	Each separator divides key ranges between subtrees, so an internal node with N separator keys must point to N + 1 child ranges.
Why can storing a high key simplify internal-node layout?	It avoids the special unpaired rightmost-pointer case by letting each child pointer travel with a corresponding key bound.
What problem do overflow pages solve in a B-tree?	They allow variable-size records to exceed a fixed page's direct payload capacity without resizing the primary page.
How are overflow pages usually linked in a B-tree?	The primary page stores the first overflow page ID, and additional overflow pages point to the next overflow page in the chain.
Why do B-tree implementations often keep only part of a large record in the primary page?	Keeping a bounded payload in-page preserves predictable free space and spills only the excess bytes to overflow pages.
Why is keeping keys sorted inside a B-tree node essential?	Binary search only works on sorted keys, so maintaining order is required for efficient lookup and correct insertion points.
How does binary search work with indirection pointers in a slotted B-tree page?	The search probes sorted cell offsets, follows each chosen offset to the cell, and compares the key stored in that cell.
What are breadcrumbs in a B-tree?	An in-memory stack of visited parent nodes and child indices collected during descent so splits or merges can be propagated upward.
Why can breadcrumbs be preferable to stored parent pointers in a B-tree?	They avoid persistent parent-pointer maintenance while still allowing the algorithm to walk back up the search path when structure changes propagate.
What is rebalancing in a B-tree?	Redistributing entries between neighboring nodes to postpone splits or merges and improve average node occupancy.
How do B*-trees differ from ordinary B-trees during overflow?	They try to redistribute entries across siblings first and, when needed, split two full nodes into three roughly two-thirds-full nodes.
What optimization becomes possible when inserted keys are monotonically increasing in a B-tree?	The engine can optimize right-edge inserts because most new entries go into the rightmost leaf and splits occur mostly on the right side.
Why is bulk loading efficient for sorted B-tree input?	The tree can be built bottom-up from leaf pages using rightmost appends, avoiding most normal split and merge work.
Why is whole-file compression usually a poor fit for mutable B-tree storage?	Updating a small part of the tree could require locating and recompressing large file regions instead of touching only affected pages.
Why is page-wise compression a practical fit for B-tree storage?	Pages can be compressed and decompressed independently during normal page reads and writes.
What is page fragmentation in a slotted B-tree page?	A state where enough logical free space exists overall, but it is broken into scattered gaps rather than one contiguous region.
Why can deletes fragment a B-tree page?	Deleting a record often removes only its cell offset, leaving the old cell bytes behind as unreachable gaps.
What does vacuum or compaction do in B-tree storage?	It rewrites pages to reclaim space from unreachable cells, restore contiguous free space, and often rewrite cells in logical order.
What are the ACID properties of a database transaction?	Atomicity, consistency, isolation, and durability.
What is the role of the page cache in a database system?	It caches disk pages in memory, stages updates there first, and reduces disk I/O.
What does it mean for a cached page to be dirty?	Its in-memory contents differ from the on-disk version and must be flushed for durability.
What is pinning in a page cache?	Keeping a page in memory so it is not evicted while it is likely to be needed.
Why are upper B-tree levels good candidates for pinning in the page cache?	They are touched by many operations but occupy only a small fraction of the tree.
Why can prefetching help a database page cache?	It loads pages before they are needed, reducing stalls during predictable access patterns such as range scans.
What is the goal of a page-replacement policy?	To evict pages that are least likely to be needed again soon.
Why is FIFO usually a poor page-replacement policy for databases?	It ignores repeated accesses, so frequently reused pages such as upper B-tree nodes can be evicted too early.
Why is LRU often better than FIFO for page replacement?	It accounts for recency of access instead of only initial load order.
Why are CLOCK-style eviction algorithms attractive in database systems?	They approximate recency with low overhead and work well in concurrent implementations.
What does LFU-based eviction try to optimize?	It keeps frequently accessed pages in cache by favoring usage frequency over simple recency.
What is a write-ahead log (WAL)?	An append-only disk log that records changes before the corresponding data pages are written back.
Why does a database need write-ahead logging?	It preserves enough information to recover committed work after a crash and to undo incomplete work.
What is a log sequence number (LSN)?	A unique monotonically increasing identifier assigned to each log record.
When can a transaction be considered committed with WAL?	Only after the log has been forced through the LSN of its commit record.
What is a checkpoint in transaction recovery?	A recovery marker showing that changes up to a certain point are sufficiently reflected on disk, reducing restart work.
Why are fuzzy checkpoints used instead of fully synchronous checkpoints?	They let pages be flushed asynchronously, avoiding long pauses while still bounding recovery work.
What is the difference between physical and logical logging?	Physical logging records page state or byte-level changes, while logical logging records operations to apply.
Why do many recovery systems combine physical redo with logical undo?	Physical redo speeds crash recovery, while logical undo works better with concurrency during normal operation.
What does a steal policy allow in buffer management?	A dirty page modified by an uncommitted transaction may be flushed to disk before that transaction commits.
What does a no-force policy allow in buffer management?	A transaction may commit before all of its modified pages have been flushed to disk.
Why does a steal/no-force design require both undo and redo support?	Uncommitted changes may already be on disk and committed changes may still be only in memory.
What is ARIES?	A recovery algorithm that uses write-ahead logging with steal/no-force buffering, repeating history during redo and then undoing incomplete transactions.
What are the three phases of ARIES recovery?	Analysis, redo, and undo.
What happens during the analysis phase of ARIES?	The system reconstructs which pages were dirty and which transactions were still in progress at the time of the crash.
What happens during the redo phase of ARIES?	The system repeats logged history to restore the database state up to the crash point.
What happens during the undo phase of ARIES?	The system rolls back the effects of incomplete transactions in reverse order.
What is serializability in transaction processing?	The guarantee that a concurrent schedule has the same result as some serial execution of the same transactions.
What is a dirty read?	Reading data written by another transaction before that transaction commits.
What is a nonrepeatable read?	Reading the same row twice in one transaction and seeing different committed values.
What is a phantom read?	Repeating the same range query in one transaction and seeing a different set of rows.
What is a lost update?	One transaction's write overwrites another transaction's write without preserving both effects.
What is snapshot isolation?	Each transaction reads from a stable snapshot and usually commits only if its writes do not conflict with concurrent committed writes.
What is optimistic concurrency control (OCC)?	A scheme where transactions run mostly without blocking and are validated for conflicts before commit.
What is multiversion concurrency control (MVCC)?	A concurrency-control approach that keeps multiple timestamped versions so readers can access older committed data while writers produce new versions.
What is two-phase locking (2PL)?	A locking protocol with a growing phase that acquires locks and a shrinking phase that releases them, without acquiring new locks after any release.
What is a deadlock in transaction processing?	A cycle where transactions wait on one another's locks and none of them can proceed.
What is the difference between a lock and a latch in a database system?	Locks protect logical data access for transactions, while latches protect in-memory physical structures such as pages during short critical sections.
What is latch crabbing in a B-tree?	A technique that acquires the child latch before releasing the parent latch so upper-level latches are held only as long as needed.
Why do Blink-Trees help concurrent B-tree access?	They use sibling links and high keys so searches can continue correctly even while splits are only partially reflected in parent pointers.
