How does a storage engine differ from a database management system?	A storage engine handles low-level data storage and retrieval, while a DBMS adds higher-level features such as schema, queries, indexes, and transactions.
Why can storage engines treat keys and values as arbitrary byte sequences?	Their meaning, sorting, and representation semantics are defined by higher-level database components rather than by the storage engine itself.
Why are pluggable storage engines useful in database systems?	They let database developers reuse existing engines and switch to engines that better fit particular workloads.
What is the best way to compare databases for a real application?	Simulate the intended workloads in a production-like environment and measure the performance and operational metrics that matter to you.
Why are long-running database tests important during evaluation?	Some performance, scalability, and operational problems appear only over time or as capacity grows.
Why can operational friendliness matter more than raw performance when choosing a database?	A database that is easier to run, debug, and recover is often more valuable than one that is faster but less reliable.
Why should database evaluation account for schema, record sizes, access patterns, and read/write rates?	Those variables determine whether the database can support the required queries, scale appropriately, and be operated safely over time.
Why can reading database source code be useful even if you are only operating the database?	It helps you interpret logs, understand configuration, debug failures, and reduce recovery time when problems occur.
Why should generic benchmarks such as YCSB be used cautiously?	Generic workloads can produce misleading conclusions unless they are tailored to real production conditions and access patterns.
Why are benchmarks useful beyond comparing database vendors?	They also help define service expectations, understand system requirements, and support capacity planning.
Why should teams test new database versions before rolling them out?	New releases can improve performance and security, but they can also introduce regressions, new bugs, or unexpected behavior.
Why is designing a storage engine more than implementing a textbook data structure?	It also requires decisions about physical layout, serialization, garbage collection, concurrency, crash safety, and integration with database semantics.
Why do storage-engine design decisions inevitably involve trade-offs?	Optimizing for one goal, such as write speed, read order, density, or simplicity, usually makes another goal harder to achieve.
Why is there no universally optimal storage engine?	Different workloads and operational priorities require different trade-offs, so no single engine is best for every use case.
Why do different storage engines end up with very different designs?	They optimize for different goals, such as read latency, write latency, storage density, or operational simplicity.
What is the job of the query optimizer in a DBMS?	It removes impossible or redundant query parts and chooses the most efficient execution plan based on statistics and data placement.
What is concurrency control in DBMS architecture?	Coordinating concurrent operations so the database preserves logical and physical integrity while still executing efficiently.
Why are main-memory DBMSs attractive?	Memory access is far faster than disk access, and programming against memory is simpler than programming against disk-oriented storage.
Why are RAM cost and volatility major limits for main-memory DBMSs?	RAM is still expensive relative to persistent storage, and its contents can be lost on crashes or power failures without extra durability measures.
How do memory-based DBMSs usually provide durability?	They write changes to a sequential log and maintain a disk-resident backup for recovery.
What is checkpointing in an in-memory DBMS?	Applying batches of log records to a backup snapshot so older log segments can be discarded and recovery can start from a newer point.
Why is an in-memory DBMS not just a disk-based DBMS with a huge cache?	In-memory systems can use different layouts, data structures, and optimizations than disk-oriented systems built around page and serialization overhead.
When is a row-oriented data layout a good fit?	When workloads usually read or write most fields of a record together, such as point queries and range scans over whole rows.
Why does row-oriented layout improve record access locality?	Values belonging to the same row are stored together, so reading one record tends to fetch its related fields in the same disk block.
When is a column-oriented data layout a good fit?	When workloads scan many rows but touch only a subset of columns, especially for analytics and aggregates.
Why can column-oriented layout outperform row-oriented layout for analytics?	It reads only needed columns and benefits from better cache utilization, compression, and vectorized processing.
Why should column-oriented stores not be confused with wide-column stores?	Wide-column stores group columns into families but still organize data for key-based access to rows within those families rather than pure analytical column scans.
What is a heap file in database storage?	A data file where records are not stored in any required key order and are typically placed in write order.
What is an index-organized table in database storage?	A table that stores the actual data records inside the index structure in key order.
How do primary and secondary indexes differ in database storage?	A primary index organizes access to the main data records, while secondary indexes provide alternative lookup paths using other fields.
What is the difference between clustered and nonclustered indexes?	A clustered index keeps data records in search-key order, while a nonclustered index points to records stored in a different order.
Why might a storage engine use the primary index as an indirection layer for secondary indexes?	It reduces the cost of updating secondary pointers when records move, at the expense of an extra lookup on reads.
What three variables distinguish many storage-engine designs?	Buffering, mutability versus immutability, and ordered versus unordered storage.
Why does ordering matter in storage-engine design?	Ordered storage supports efficient range scans, while unordered or insertion-order storage can simplify and speed writes.
Why are binary search trees a poor fit for disk-based storage?	Their low fanout creates tall trees with many pointer traversals and frequent rebalancing, which leads to too many costly disk accesses and pointer updates.
Why does a disk-friendly tree structure need high fanout and low height?	Higher fanout stores more keys per node and reduces the number of levels and disk seeks needed during lookup.
What is fanout in a tree-based storage structure?	The maximum number of child pointers a node can have.
How are fanout and tree height related?	Higher fanout lowers tree height because each node covers a larger part of the key space.
Why are B-trees well suited for disk-based storage?	They keep many keys in each node, reduce tree height, improve locality, and minimize the number of disk seeks per lookup.
Why can B-trees support both point queries and range scans efficiently?	Keys are stored in sorted order, so the tree can both locate exact items quickly and iterate through key ranges in order.
What is a B+-tree?	A B-tree variant that stores values only in leaf nodes while internal nodes store only separator keys and child pointers.
Why are B+-trees commonly treated as the default form of B-trees?	They are the widespread practical variant used by many storage engines and textbooks often refer to them simply as B-trees.
Why can leaf-level sibling pointers be useful in B-trees?	They let range scans continue across neighboring leaves without repeatedly going back up to parent nodes.
Why does B-tree lookup require relatively few disk seeks even for very large datasets?	Each node contains many keys, so one disk read narrows the search space by a large factor.
What causes a B-tree node split?	An insertion overflows a node beyond its maximum capacity.
What happens during a B-tree split?	A new sibling node is allocated, roughly half the entries move to it, and a separator key is promoted to the parent.
Why can a split propagate all the way to the root in a B-tree?	If the parent is already full, promoting the separator overflows that parent too, forcing recursive splits upward.
What happens when a B-tree root splits?	A new root is created above the old root and its new sibling, increasing the tree height by one.
What causes a B-tree node merge?	A deletion leaves neighboring nodes underfull enough that their contents fit into a single node.
What happens during a B-tree merge?	Entries from one sibling are copied into the other, the parent's separator or pointer is removed, and one node disappears.
Why can merges propagate upward in a B-tree?	Removing a separator or pointer from a parent can make that parent underfull as well.
Why can B-trees tolerate node occupancy well below 100%?	Keeping slack space reduces the frequency of splits and merges while still preserving strong lookup performance.
Why can sibling links be useful in a B-tree?	They let the engine move to neighboring nodes on the same level without first climbing back up through parent nodes.
What trade-off comes with storing sibling links in a B-tree?	Splits and merges become more complex because neighboring pages' sibling pointers must also be updated.
Why does a B-tree internal node have one more child pointer than separator keys?	Each separator divides key ranges between subtrees, so an internal node with N separator keys must point to N + 1 child ranges.
Why can storing a high key simplify internal-node layout?	It avoids the special unpaired rightmost-pointer case by letting each child pointer travel with a corresponding key bound.
What problem do overflow pages solve in a B-tree?	They allow variable-size records to exceed a fixed page's direct payload capacity without resizing the primary page.
How are overflow pages usually linked in a B-tree?	The primary page stores the first overflow page ID, and additional overflow pages point to the next overflow page in the chain.
Why do B-tree implementations often keep only part of a large record in the primary page?	Keeping a bounded payload in-page preserves predictable free space and spills only the excess bytes to overflow pages.
Why is keeping keys sorted inside a B-tree node essential?	Binary search only works on sorted keys, so maintaining order is required for efficient lookup and correct insertion points.
How does binary search work with indirection pointers in a slotted B-tree page?	The search probes sorted cell offsets, follows each chosen offset to the cell, and compares the key stored in that cell.
What are breadcrumbs in a B-tree?	An in-memory stack of visited parent nodes and child indices collected during descent so splits or merges can be propagated upward.
Why can breadcrumbs be preferable to stored parent pointers in a B-tree?	They avoid persistent parent-pointer maintenance while still allowing the algorithm to walk back up the search path when structure changes propagate.
What is rebalancing in a B-tree?	Redistributing entries between neighboring nodes to postpone splits or merges and improve average node occupancy.
How do B*-trees differ from ordinary B-trees during overflow?	They try to redistribute entries across siblings first and, when needed, split two full nodes into three roughly two-thirds-full nodes.
What optimization becomes possible when inserted keys are monotonically increasing in a B-tree?	The engine can optimize right-edge inserts because most new entries go into the rightmost leaf and splits occur mostly on the right side.
Why is bulk loading efficient for sorted B-tree input?	The tree can be built bottom-up from leaf pages using rightmost appends, avoiding most normal split and merge work.
Why is whole-file compression usually a poor fit for mutable B-tree storage?	Updating a small part of the tree could require locating and recompressing large file regions instead of touching only affected pages.
Why is page-wise compression a practical fit for B-tree storage?	Pages can be compressed and decompressed independently during normal page reads and writes.
What is page fragmentation in a slotted B-tree page?	A state where enough logical free space exists overall, but it is broken into scattered gaps rather than one contiguous region.
Why can deletes fragment a B-tree page?	Deleting a record often removes only its cell offset, leaving the old cell bytes behind as unreachable gaps.
What does vacuum or compaction do in B-tree storage?	It rewrites pages to reclaim space from unreachable cells, restore contiguous free space, and often rewrite cells in logical order.
What are the ACID properties of a database transaction?	Atomicity, consistency, isolation, and durability.
What is the role of the page cache in a database system?	It caches disk pages in memory, stages updates there first, and reduces disk I/O.
What does it mean for a cached page to be dirty?	Its in-memory contents differ from the on-disk version and must be flushed for durability.
What is pinning in a page cache?	Keeping a page in memory so it is not evicted while it is likely to be needed.
Why are upper B-tree levels good candidates for pinning in the page cache?	They are touched by many operations but occupy only a small fraction of the tree.
Why can prefetching help a database page cache?	It loads pages before they are needed, reducing stalls during predictable access patterns such as range scans.
What is the goal of a page-replacement policy?	To evict pages that are least likely to be needed again soon.
Why is FIFO usually a poor page-replacement policy for databases?	It ignores repeated accesses, so frequently reused pages such as upper B-tree nodes can be evicted too early.
Why is LRU often better than FIFO for page replacement?	It accounts for recency of access instead of only initial load order.
Why are CLOCK-style eviction algorithms attractive in database systems?	They approximate recency with low overhead and work well in concurrent implementations.
What does LFU-based eviction try to optimize?	It keeps frequently accessed pages in cache by favoring usage frequency over simple recency.
What is a write-ahead log (WAL)?	An append-only disk log that records changes before the corresponding data pages are written back.
Why does a database need write-ahead logging?	It preserves enough information to recover committed work after a crash and to undo incomplete work.
What is a log sequence number (LSN)?	A unique monotonically increasing identifier assigned to each log record.
When can a transaction be considered committed with WAL?	Only after the log has been forced through the LSN of its commit record.
What is a checkpoint in transaction recovery?	A recovery marker showing that changes up to a certain point are sufficiently reflected on disk, reducing restart work.
Why are fuzzy checkpoints used instead of fully synchronous checkpoints?	They let pages be flushed asynchronously, avoiding long pauses while still bounding recovery work.
What is the difference between physical and logical logging?	Physical logging records page state or byte-level changes, while logical logging records operations to apply.
Why do many recovery systems combine physical redo with logical undo?	Physical redo speeds crash recovery, while logical undo works better with concurrency during normal operation.
What does a steal policy allow in buffer management?	A dirty page modified by an uncommitted transaction may be flushed to disk before that transaction commits.
What does a no-force policy allow in buffer management?	A transaction may commit before all of its modified pages have been flushed to disk.
Why does a steal/no-force design require both undo and redo support?	Uncommitted changes may already be on disk and committed changes may still be only in memory.
What is ARIES?	A recovery algorithm that uses write-ahead logging with steal/no-force buffering, repeating history during redo and then undoing incomplete transactions.
What are the three phases of ARIES recovery?	Analysis, redo, and undo.
What happens during the analysis phase of ARIES?	The system reconstructs which pages were dirty and which transactions were still in progress at the time of the crash.
What happens during the redo phase of ARIES?	The system repeats logged history to restore the database state up to the crash point.
What happens during the undo phase of ARIES?	The system rolls back the effects of incomplete transactions in reverse order.
What is serializability in transaction processing?	The guarantee that a concurrent schedule has the same result as some serial execution of the same transactions.
What is a dirty read?	Reading data written by another transaction before that transaction commits.
What is a nonrepeatable read?	Reading the same row twice in one transaction and seeing different committed values.
What is a phantom read?	Repeating the same range query in one transaction and seeing a different set of rows.
What is a lost update?	One transaction's write overwrites another transaction's write without preserving both effects.
What is snapshot isolation?	Each transaction reads from a stable snapshot and usually commits only if its writes do not conflict with concurrent committed writes.
What is optimistic concurrency control (OCC)?	A scheme where transactions run mostly without blocking and are validated for conflicts before commit.
What is multiversion concurrency control (MVCC)?	A concurrency-control approach that keeps multiple timestamped versions so readers can access older committed data while writers produce new versions.
What is two-phase locking (2PL)?	A locking protocol with a growing phase that acquires locks and a shrinking phase that releases them, without acquiring new locks after any release.
What is a deadlock in transaction processing?	A cycle where transactions wait on one another's locks and none of them can proceed.
What is the difference between a lock and a latch in a database system?	Locks protect logical data access for transactions, while latches protect in-memory physical structures such as pages during short critical sections.
What is latch crabbing in a B-tree?	A technique that acquires the child latch before releasing the parent latch so upper-level latches are held only as long as needed.
Why do Blink-Trees help concurrent B-tree access?	They use sibling links and high keys so searches can continue correctly even while splits are only partially reflected in parent pointers.
What problem does copy-on-write solve in B-tree implementations?	It preserves consistency for concurrent readers and writers by modifying copied pages instead of overwriting live ones in place.
How does copy-on-write update a B-tree?	It copies the pages on the modified path, updates the copies, and atomically switches the topmost pointer to the new hierarchy.
Why can readers run without synchronization in a copy-on-write B-tree?	They read immutable old pages that are never modified in place.
What is the main trade-off of copy-on-write B-trees?	They simplify concurrency and crash safety but require extra copying work and temporary extra space.
Why can LMDB avoid a page cache and write-ahead log?	It uses memory mapping plus copy-on-write, so committed structure changes are published atomically through root switching.
What is a lazy B-tree in this chapter's sense?	A B-tree variant that buffers updates in auxiliary in-memory structures instead of applying each change to tree pages immediately.
Why can buffering updates improve B-tree write performance?	It batches multiple same-node changes together, reducing repeated random writes and structural modifications.
How does an LA-Tree differ from per-node buffering schemes?	It buffers updates for entire subtrees and cascades them downward in batches rather than updating nodes individually right away.
What is the main idea behind FD-Trees?	Keep a small mutable head B-tree for recent writes and store older data in larger immutable sorted runs.
Why do FD-Trees reduce random write costs?	Most writes first hit the small head tree, and bulk propagation to lower levels happens through sequential merges into immutable runs.
How are deletes usually represented in FD-Trees?	As tombstones that hide older values in lower levels until compaction or propagation removes them.
What is the main idea behind a Bw-Tree?	Represent a logical node as a base node plus an append-only chain of delta records, with an in-memory mapping table pointing to the newest version.
How do Bw-Trees reduce write amplification?	Updates append small delta records instead of rewriting whole pages for every change.
Why do Bw-Trees use a mapping table?	It lets the tree update logical node pointers atomically without rewriting parent pages on every modification.
How do Bw-Trees tame concurrency without latches?	They use compare-and-swap to install new mapping-table pointers to appended delta nodes atomically.
What is the main read-time trade-off in a Bw-Tree?	Readers may need to traverse and apply a chain of deltas to reconstruct the current logical node state.
What is consolidation in a Bw-Tree?	Merging a logical node's base page and accumulated delta chain into a new compact base node.
How do split and merge operations work differently in a Bw-Tree than in a classic B-tree?	They proceed through multistep structural modification operations recorded with special delta nodes rather than in-place page rewrites.
What is the main trade-off between mutable and immutable storage structures?	Mutable structures optimize reads by updating records in place, while immutable structures optimize writes by appending new records and reconciling duplicates later.
Why are immutable storage structures often safer for concurrency?	Once written, their contents do not change, so readers can access shared structures without worrying about in-place modification.
What problem does log-structured storage primarily try to reduce?	Write amplification from many small random updates by turning them into sequential append-like writes.
What is the main idea behind an LSM tree?	Buffer writes in memory, flush them to immutable disk tables, and merge those tables later through compaction.
Why are LSM trees often write-optimized relative to B-trees?	Writes usually avoid locating and rewriting existing on-disk records, instead appending buffered data to new immutable files.
Why are reads more expensive in LSM trees than in B-trees?	Relevant data may be spread across multiple memory- and disk-resident components and must be merged and reconciled.
What is a memtable in an LSM tree?	The mutable in-memory table that receives current reads and writes before its contents are flushed to disk.
Why does an LSM tree still need a write-ahead log?	Memtable contents are only in memory until flush, so the log is needed for durability and crash recovery.
How are deletes usually represented in LSM trees?	As tombstones that mark older values as deleted until compaction can safely remove the shadowed records.
Why can't an LSM tree implement delete by simply removing a key from the memtable?	Older copies may still exist in disk tables, so the value could reappear unless a tombstone suppresses it.
What is reconciliation in an LSM tree?	The process of resolving multiple versions of the same key across components and keeping only the newest live value.
What is compaction in an LSM tree?	A maintenance process that merges immutable tables, discards shadowed or deleted data, and rewrites the merged result into new tables.
Why does compaction create write amplification in LSM trees?	Data may be rewritten multiple times as it moves through levels during maintenance.
Why must tombstones be preserved during compaction until the right point?	Dropping them too early can resurrect older shadowed values still present in other tables.
What is leveled compaction?	A strategy that organizes tables into levels with size targets and merges overlapping ranges into the next level.
What is size-tiered compaction?	A strategy that groups and merges tables of similar size rather than enforcing nonoverlapping levels as strictly as leveled compaction.
What are the three main amplification trade-offs in LSM trees?	Read amplification, write amplification, and space amplification.
What does the RUM conjecture say about storage structures?	You cannot simultaneously minimize read cost, update cost, and memory overhead; improving two tends to worsen the third.
Why are SSTables a natural fit for LSM trees?	They can be written sequentially once, searched efficiently, and merged later without in-place updates.
How does an LSM merge iterator work?	It keeps the current head item from each source in priority order and repeatedly emits the smallest key while refilling from the corresponding source.
Why can Bloom filters reduce read amplification in LSM trees?	They let the system skip tables that definitely do not contain a searched key.
What guarantee does a Bloom filter provide?	It may return false positives, but it does not return false negatives.
What is the main idea behind Bitcask?	Append key-value records directly to log files and keep an in-memory directory pointing to the latest value for each key.
What is the main limitation of Bitcask's design?	It is excellent for point lookups but does not support efficient range scans because data is not kept in key order.
What is the main idea behind WiscKey?	Keep keys in an LSM-tree index while storing values separately in append-only value logs.
Why can WiscKey reduce compaction costs?	Compaction mostly rewrites smaller keys and pointers instead of repeatedly rewriting large values.
What is the main trade-off of WiscKey?	It reduces write and compaction costs for large values but makes range scans more random-I/O heavy because values are not stored in sorted order.
What makes distributed systems fundamentally harder than single-node systems?	Independent processes communicate over unreliable networks, so latency, message loss, reordering, and partial failure all affect correctness.
Why does concurrency already resemble a distributed-systems problem?	Multiple execution contexts can observe and update shared state in different orders, so outcomes become nondeterministic unless coordination constrains them.
Why is adding a shared database not enough to make distributed state simple?	Processes still depend on slow, failure-prone communication and cannot always tell whether remote state is current, delayed, or unavailable.
Why is it dangerous to assume the network is reliable in a distributed system?	Messages or responses can be delayed, dropped, or interrupted even after a connection succeeds.
Why should distributed systems never assume zero latency?	Every remote call pays transport, software-stack, and processing costs, so remote communication is never instantaneous.
Why is infinite bandwidth a bad assumption in distributed systems?	As load, message size, or node count grows, links can saturate and slow or block useful work.
Why is remote processing time part of the distributed-systems problem?	A remote node must schedule and execute work after receiving a message, so end-to-end latency includes both communication and processing delays.
What is backpressure in a distributed system?	A way to slow producers when consumers cannot keep up, preventing queues from growing without bound and amplifying overload.
Why can simply enlarging queues make a distributed system worse?	Queued work still waits unprocessed, so bigger queues can increase latency without improving throughput.
Why is synchronized physical time hard to rely on in distributed systems?	Clocks on different machines drift, so timestamps alone are unsafe for exact ordering or coordination unless timing bounds are known.
Why should local and remote execution not be treated as interchangeable?	Remote execution adds latency, serialization, failure modes, and consistency concerns that local calls do not have.
What is a partial failure in a distributed system?	A failure where some components or links are broken while the rest of the system continues running.
What is a cascading failure in a distributed system?	A failure that spreads because trouble in one part increases load or instability in other parts.
How do backoff and jitter help during distributed-system failures?	Backoff spaces out retries, and jitter randomizes them so many clients do not retry in lockstep.
What is a circuit breaker in distributed systems?	A mechanism that detects repeated failures and temporarily stops sending work to a failing dependency so it can recover.
Why should distributed systems validate exchanged data with checksums or similar mechanisms?	To detect corruption before bad data is propagated and replicated to other nodes.
What do acknowledgments add to message passing?	They let the sender learn that the receiver got a specific message, even though either the message or the acknowledgment may still be lost.
Why are retransmits alone not enough for reliable distributed operations?	If the sender cannot tell whether an earlier attempt already succeeded, retries can create duplicates and unintended side effects.
Why is idempotence important in distributed systems?	It makes repeated execution safe when retries or duplicate deliveries occur.
Why is exactly-once delivery usually really about exactly-once processing?	Messages may be transmitted more than once under the hood, but the important guarantee is that duplicates are ignored so the operation's effect happens only once.
What does the Two Generals' Problem show?	Two parties cannot achieve guaranteed agreement over an unreliable asynchronous channel because there is always uncertainty about the last acknowledgment.
What does the FLP impossibility result show?	In a fully asynchronous system, no deterministic consensus algorithm can guarantee termination if even one process may crash.
What is a partially synchronous distributed system?	One that behaves asynchronously at times but usually satisfies enough timing assumptions for practical coordination algorithms to work.
What is a crash-stop failure model?	A model where a failed process stops taking further steps and does not recover within the current execution.
What is a crash-recovery failure model?	A model where a process may fail, later recover, and continue participating using durable state.
What is an omission fault?	A failure where a process or link skips expected actions, such as sending, receiving, or exposing some messages or results.
What is a Byzantine fault?	A failure where a process continues acting but produces arbitrary or contradictory behavior instead of following the algorithm.
Why do many practical distributed algorithms assume crash faults instead of Byzantine faults?	Crash-fault models are easier to reason about and implement, while Byzantine tolerance is much more complex and expensive.
Why is failure detection hard in asynchronous distributed systems?	Without timing guarantees, a process that crashed can look the same as a process or network path that is merely very slow.
What is the practical role of a failure detector in a distributed system?	To identify failed or unreachable processes so algorithms can stop relying on them and continue making progress without violating safety.
What is the difference between a failed process and a suspected process?	A failed process has stopped executing, while a suspected process is believed to be faulty or unreachable but may still be alive.
What trade-off does failure detection make between false positives and false negatives?	Marking processes failed quickly improves responsiveness but risks excluding live processes; waiting longer reduces false positives but delays recovery from real failures.
Why can't a practical failure detector be both perfectly accurate and maximally fast?	In asynchronous systems, delay is indistinguishable from failure, so faster suspicion necessarily increases the risk of false positives.
Why are failure detectors important for consensus and atomic broadcast?	They augment the asynchronous model with suspicion information that lets protocols make progress despite crashes or unreachable members.
What is the difference between ping-based and heartbeat-based failure detection?	Pings actively query peers and wait for replies, while heartbeats have processes proactively announce that they are still alive.
Why do heartbeat-and-timeout detectors depend heavily on timeout tuning?	A timeout that is too short creates false suspicions under delay, while a timeout that is too long delays detection of real failures.
How does a timeout-free heartbeat detector avoid direct timeout assumptions?	It counts and propagates heartbeat information through fair paths, letting applications compare heartbeat counters rather than relying on fixed deadlines.
Why can propagated heartbeat paths detect liveness despite a faulty direct link?	Other processes may still receive and relay the heartbeat, showing that the target is alive through an indirect path.
What is the main idea of outsourced heartbeats?	When a direct ping fails, a process asks other peers to probe the suspect and report whether they can reach it.
Why do outsourced heartbeats improve failure-detection decisions?	They combine direct and indirect reachability evidence, reducing the chance that one bad link is mistaken for a process failure.
What is the main idea of a phi-accrual failure detector?	It reports a continuous suspicion level based on heartbeat arrival statistics instead of making an immediate binary up/down decision.
How does a phi-accrual failure detector adapt to changing network conditions?	It uses recent heartbeat arrival samples to adjust the suspicion scale, so slower but normal conditions do not look identical to sudden failure.
How does gossip-style failure detection form a more reliable cluster view?	Members periodically exchange heartbeat counters and timestamps with random peers, merging multiple local views into a broader view of member liveness.
Why can gossip detect failure despite a broken link between two nodes?	Heartbeat information can still propagate through other members, so one link failure does not necessarily hide a live process.
What is the trade-off of gossip-style failure detection?	It adds more messages to spread state, but produces a more reliable aggregate view than a single direct check.
What is the main idea behind FUSE-style failure detection?	It propagates individual failure information by turning it into group failure: members that detect a failed peer stop responding themselves.
How does FUSE use quiescence to propagate failure?	When a member detects a peer failure, it stops answering pings itself, so the absence of responses spreads the failure signal through the group.
What is a downside of FUSE-style group failure propagation?	A link failure or partition around one member can be amplified into a group failure, which may or may not match the application's desired semantics.
Why do distributed algorithms often use a leader?	A leader reduces coordination overhead by letting one process coordinate steps, maintain shared ordering, or drive decisions instead of requiring every participant to contact every other participant.
What are the key liveness and safety goals of leader election?	Liveness means election eventually completes and a leader exists most of the time; safety means at most one leader exists for the same role at a time.
What is split brain in leader election?	A failure mode where two or more partitions independently elect leaders that serve the same role without knowing about each other.
Why is leader election different from distributed locking?	With a lock, others mainly need eventual access; with leader election, participants need to know which process has the special coordination role.
Why are long-lived leaders usually preferred?	Stable leadership reduces repeated synchronization, election overhead, and state handoff between processes.
What is the main bottleneck risk of leader-based coordination?	The leader can become a throughput, latency, or availability bottleneck if too much work must pass through one process.
How can systems reduce the bottleneck risk of a single leader?	Partition the data or work into independent replica sets, each with its own leader.
Why must leader election be paired with failure detection?	The system needs to detect when the current leader has crashed or become unreachable so another leader can be elected.
What is the core idea of the bully leader election algorithm?	Processes have unique ranks, and the highest-ranked reachable process becomes the leader.
How does a process start a bully election?	It sends election messages to higher-ranked processes to check whether any of them are alive and should take over.
When can a process declare itself leader in the bully algorithm?	When it finds no active process with a higher rank and then notifies lower-ranked processes of the result.
Why can the bully algorithm produce split brain under network partitions?	Each partition may see no higher-ranked reachable leader and independently elect a leader.
Why can unstable high-ranked nodes be harmful in the bully algorithm?	They repeatedly win elections and then fail, causing recurring reelections and poor liveness.
How can host-quality metrics improve bully-style leader election?	The election can consider stability or quality, not only rank, reducing preference for unstable high-ranked nodes.
How does next-in-line failover optimize bully-style election?	The old leader publishes alternative leaders, so a detector can contact the highest-ranked alternative first instead of running a full election.
What is the candidate/ordinary optimization for leader election?	It splits processes into candidate and ordinary sets, so only candidate nodes can become leader and ordinary nodes query candidates during election.
How does the candidate/ordinary optimization reduce election messages?	Ordinary nodes contact only the candidate set rather than every higher-ranked process.
What is the main idea of the invitation leader election algorithm?	Processes start as leaders of their own groups and invite other groups to merge instead of competing by rank.
What is the main benefit of invitation-style election?	It can create and merge process groups without restarting a full election from scratch.
What topology assumption does the ring election algorithm make?	Processes are arranged in a logical ring and know their predecessor and successor relationships.
How does the ring algorithm collect leader candidates?	An election message traverses the ring, and each reachable process contributes its identity or updates the current highest-ranked identity.
How is the leader chosen in the ring algorithm?	After the election message completes a ring traversal, the highest-ranked live process seen during traversal is selected.
Why can the ring algorithm still suffer split brain?	If the ring is partitioned, each partition may complete its own traversal and elect its own leader.
Why is a cluster-wide majority important for avoiding split brain?	A majority quorum prevents two disjoint partitions from both obtaining enough votes to safely elect independent leaders.
How do consensus algorithms use leaders differently from simple election algorithms?	They may allow temporary competing leaders but use quorums, terms, or ballots to detect conflicts and preserve safety.
Why is allowing multiple temporary leaders sometimes a performance optimization?	The system can proceed speculatively and resolve conflicts through later quorum checks instead of blocking on perfect leader uniqueness upfront.
How does Raft detect an out-of-date leader?	A leader that observes a higher term learns that a newer leader may exist and updates or steps down accordingly.
Why is leader identity alone not enough for safe coordination?	Processes also need to know whether their local knowledge of the leader is still current despite failures, delays, and elections.
Why is replication fundamental to fault tolerance?	Replication removes single points of failure by keeping multiple copies of data so the system can continue serving requests when some components fail.
Why is replication especially valuable in multidatacenter deployments?	It improves availability through geographic redundancy and can reduce latency by placing data closer to clients.
Why is keeping replicas perfectly in sync expensive?	Atomically updating multiple copies is equivalent to solving consensus, which adds significant coordination cost.
What three events matter most when reasoning about replication visibility?	Write, replica update, and read.
What does CAP mean by availability?	Every request to a nonfailing node eventually receives a response.
What does CAP mean by consistency?	Operations appear atomic and linearizable rather than merely preserving database invariants.
Why can't a distributed system guarantee both strong consistency and availability during a network partition?	Partitions prevent some nodes from coordinating, so the system must either reject some requests to preserve consistency or answer with potentially inconsistent data.
What is a CP system in CAP terms?	A system that prefers rejecting requests over serving potentially inconsistent data during partitions.
What is an AP system in CAP terms?	A system that continues serving requests during partitions even if some responses may be inconsistent.
What does PACELC add beyond CAP?	It says that even when there is no partition, distributed systems still trade consistency against latency.
Why should CAP be used carefully?	It concerns partitions rather than all failure types, and its formal definitions of consistency and availability differ from the looser meanings often used in practice.
Why isn't partition tolerance really a tunable choice like consistency versus availability?	Real networks can partition whether we want them to or not, so systems must tolerate partitions instead of trading them away.
What is harvest in a distributed query system?	The completeness of a response, such as returning 99 of 100 rows instead of failing the entire query.
What is yield in a distributed query system?	The proportion of requests that complete successfully out of all attempted requests.
Why is trading harvest for yield sometimes useful?	Returning partial but useful results can keep more requests successful during failures instead of failing them completely.
What is a safe register?	A register where reads concurrent with a write may return arbitrary values from the register's range.
What is a regular register?	A register where a read returns either the most recent completed write or a value from a write concurrent with that read.
What is an atomic register?	A register that guarantees linearizability, so writes appear to take effect at a single instant.
Why are consistency models useful in distributed systems?	They define which read and write histories are allowed, making it possible to reason about visibility despite replication and concurrency.
Why is strict consistency mostly a theoretical model?	It requires instant global visibility based on real time, which is impossible in real distributed systems.
What is linearizability?	Each operation appears to take effect exactly once at some point between invocation and completion, consistent with real-time order.
What does linearizability prevent once a write becomes visible?	It prevents later reads from moving backward to older values, so stale reads are disallowed after the write's linearization point.
Why is linearizability expensive in distributed systems?	It requires coordination and ordering, typically through consensus or similarly strong synchronization.
What is the main difference between sequential consistency and linearizability?	Sequential consistency preserves per-process program order without requiring real-time order, while linearizability also respects real-time ordering.
Why can sequential consistency still allow stale reads?	Operations can become visible later than their completion time as long as all processes observe a single consistent global order.
What is causal consistency?	All processes must observe causally related operations in the same order, while unrelated concurrent writes may be seen in different orders.
Why can causally unrelated writes be observed in different orders under causal consistency?	No happened-before relationship constrains them, so different replicas may receive and expose them in different orders.
What is the main implementation idea behind causal consistency?	Track dependencies with logical metadata so an update becomes visible only after the updates it depends on have already been applied.
What are vector clocks used for in replicated systems?	To capture partial causal order, detect divergent histories, and help reconcile conflicting versions.
Why don't vector clocks fully solve conflict resolution by themselves?	They can detect that versions conflict, but application-specific logic is often still needed to decide how to merge them.
What are session consistency models for?	They describe the guarantees a single client should observe while interacting with a replicated system.
What is read-your-writes consistency?	A client's later reads must be able to observe its own earlier writes.
What are monotonic reads?	Once a client has seen a value, later reads by that client cannot return an older value.
What are monotonic writes?	Writes from the same client must become visible to others in the order the client issued them.
What are writes-follow-reads?	A client's writes must be ordered after the writes whose effects that client has already observed.
What is PRAM or FIFO consistency?	A model where writes from the same process are seen in order, but writes from different processes may be observed in different orders.
What is eventual consistency?	If no new updates occur, all replicas eventually converge to the latest reconciled value.
What are N, W, and R in tunable consistency?	N is replication factor, W is required write acknowledgments, and R is required read responses.
Why does the condition R + W > N matter?	It guarantees overlap between read and write sets, so reads can observe the most recent completed write.
How does increasing W or R affect a tunably consistent system?	It improves consistency but increases latency and requires more replicas to be available.
Why don't quorum reads and writes automatically guarantee monotonic reads?	Incomplete writes may reach only some replicas, so different quorum reads can still alternate between old and newer values.
How do witness replicas reduce storage cost without giving up quorum behavior?	Some replicas record that a write occurred instead of permanently storing full copies, and can temporarily hold real data when a copy replica is unavailable or times out.
What is strong eventual consistency?	Replicas may diverge temporarily, but once all updates are delivered, they deterministically converge to the same correct state.
Why are CRDTs useful in eventually consistent systems?	They allow replicas to apply updates independently and still merge their states without coordination conflicts.
What is the core idea of grow-only and PN-counter CRDTs?	A grow-only counter lets each node increment only its own slot and merges by taking per-slot maxima; a PN-counter adds separate positive and negative counters so decrements can be represented by subtracting the negative total from the positive total.
What is a last-write-wins register?	A register that resolves conflicting writes by keeping the value with the greatest timestamp.
What is the main trade-off of last-write-wins conflict resolution?	It simplifies convergence but can discard concurrent updates that might both matter to the application.
Why is direct one-to-many broadcast often a poor fit for large clusters?	It depends heavily on one sender, scales poorly with cluster size, and requires the broadcaster and each recipient to overlap in availability.
What problem does anti-entropy solve in replicated systems?	It repairs replica divergence after primary delivery fails or some updates are missed.
Why is full pairwise dataset comparison usually too expensive for replica repair?	It requires scanning and exchanging very large amounts of data even when only a small subset has diverged.
What is read repair?	A coordinator detects inconsistent replica responses during a read and sends missing updates to lagging replicas.
Why is read repair naturally scoped to hot data?	It only repairs the records that are actually being read rather than the whole dataset.
What is the trade-off of blocking read repair?	It improves read monotonicity but hurts availability and latency because the read waits for repairs to complete.
What is the trade-off of asynchronous read repair?	It returns results faster but does not immediately guarantee repaired replicas for later reads.
Why can quorum reads still return consistent results without repairing every replica?	It is enough to consult a sufficiently overlapping subset of replicas rather than every copy in the system.
What is a digest read?	A read where some replicas return only a hash of their local data so the coordinator can cheaply detect divergence.
Why are digest reads useful on the happy path?	Matching digests let the coordinator confirm replicas agree without fetching full data from every replica.
What must happen when digest reads disagree?	The coordinator must issue full reads to the differing replicas, reconcile the values, and repair lagging nodes.
What is hinted handoff?	A write-side repair technique where a coordinator or replica temporarily stores a hint for a write that a target replica missed.
Why aren't hinted writes normally counted as full replicas?	Hints are temporary repair records and are not meant to serve normal reads like real replicas.
What is sloppy quorum?	A quorum technique that uses substitute healthy nodes when the intended replicas are unavailable.
What trade-off do sloppy quorums make?	They improve availability and durability during failures but can weaken consistency because substitute nodes are not the intended replicas.
Why can sloppy quorums break immediate read-your-write behavior?	A later read from the original replica set may miss a recent write that was temporarily stored on substitute nodes.
What is the main purpose of Merkle trees in anti-entropy?	To compare compact hierarchical hashes instead of whole datasets and quickly narrow repair to inconsistent ranges.
How does a Merkle tree help localize replica differences?	Mismatched hashes higher in the tree identify which subtree to descend into until the divergent data range is isolated.
What is the main trade-off in Merkle tree design?	Finer trees give more precise repair ranges but increase tree size and maintenance cost.
What do bitmap version vectors track?	Which writes from each peer a node has seen, including gaps where some updates are missing.
Why are bitmap version vectors useful for anti-entropy?	They let replicas identify specific missing writes instead of rechecking broad data ranges.
What is a downside of bitmap version vectors?	A long-down node can prevent aggressive log truncation because peers must preserve information needed to catch it up.
What is the main idea of gossip dissemination?	Nodes randomly exchange hot information with peers so updates spread cooperatively through the cluster.
What do infective, susceptible, and removed mean in gossip protocols?	Infective nodes actively spread an update, susceptible nodes have not yet seen it, and removed nodes stop propagating it.
Why is redundancy essential to gossip?	Duplicate paths and repeated delivery make propagation robust even when some nodes or links fail.
What does gossip latency measure?	How long the system takes to approach convergence on an update.
How do gossip protocols decide when to stop spreading a message?	Nodes lose interest after enough duplicate sightings or according to a probability function that suggests the update is already widespread.
Why are pure gossip protocols not message-optimal?	They intentionally send redundant messages to gain robustness and probabilistic delivery guarantees.
What is the weakness of dissemination over only a spanning tree?	A single broken link can disconnect an entire subtree from the broadcast.
Why combine tree-based broadcast with gossip?	The tree keeps normal dissemination efficient, while gossip helps recover from failures and repair connectivity.
What is the main idea of Plumtree?	Use an eager spanning-tree-like broadcast for efficiency and lazy gossip links for recovery and repair.
How does Plumtree's lazy push help reliability?	Nodes advertise message IDs broadly, and peers that missed a full message can request it and help heal the dissemination tree.
Why can Plumtree tend toward low-latency dissemination trees over time?	Faster responders are more likely to become part of the eager broadcast structure.
Why do gossip systems often maintain only partial views of the cluster?	Keeping a full membership view is expensive and unnecessary when reliable dissemination can work from overlapping subsets.
What is the difference between active and passive views in HyParView?	Active view peers are used for dissemination, while passive view peers are reserves for replacing failed active neighbors.
Why does HyParView use shuffle operations?	To refresh partial views and exchange membership information without requiring every node to know the whole cluster.
How does HyParView recover from an active-peer failure?	The node removes the failed peer and promotes a replacement from its passive view if a connection can be established.
What is the main benefit of hybrid gossip approaches like HyParView and Plumtree?	They reduce normal message cost while preserving strong failure recovery through broader fallback dissemination.
How do hinted handoff, read repair, Merkle trees, and bitmap version vectors optimize different aspects of repair?	Hinted handoff targets recent missed writes, read repair targets queried data, Merkle trees target inconsistent ranges, and bitmap vectors target specific missing events.
Why are multipartition transactions harder than node-local transactions?	They require coordination across servers so all participants either commit durably or roll back together.
Why must distributed transactions be recoverable?	If execution fails or aborts, every participant must be able to roll back so the database returns to its prior consistent state.
Why don't atomic commitment protocols tolerate Byzantine failures?	They assume participants honestly report commit or abort decisions rather than lying or choosing arbitrary outcomes.
What does a transaction manager do in a distributed database?	It schedules, coordinates, executes, and tracks transactions so local visibility rules line up with global commit decisions.
What are the two phases of two-phase commit?	A prepare phase that collects votes and a commit-or-abort phase that finalizes the decision.
What happens during the prepare phase of two-phase commit?	The coordinator asks cohorts whether they can commit their part of the transaction and records their votes durably.
Why does two-phase commit hurt availability?	A single unavailable participant can prevent a distributed transaction from committing.
Why do cohorts log their decisions durably in two-phase commit?	So they can recover their state and learn or replay the final outcome after failures.
Why is a cohort that voted yes still uncertain after a coordinator failure?	It promised not to abort on its own, but may not know whether the coordinator later committed or aborted.
Why is two-phase commit called a blocking protocol?	If the coordinator fails at the wrong time, cohorts can be forced to wait indefinitely for the final decision.
What is the worst coordinator failure point in two-phase commit?	After collecting yes votes but before notifying cohorts of the final outcome.
Why can cohorts safely replay a commit decision learned from peers or logs?	Because a commit in two-phase commit is unanimous: if one cohort was told to commit, all cohorts must commit.
What extra phase does three-phase commit add?	A prepare phase between voting and final commit so cohorts can distinguish safer recovery states.
How does three-phase commit try to avoid blocking on coordinator failure?	It uses an extra state plus timeouts so cohorts can deterministically move toward abort or commit depending on how far the protocol progressed.
Why is three-phase commit less practical than it sounds?	It assumes a synchronous environment without partition-like communication failures and can still split decisions under partitions.
What is the main failure risk of three-phase commit during a network partition?	Some cohorts may decide to commit while others time out and abort, creating split brain.
Why is two-phase commit still widely used despite blocking?	It is conceptually simple, has low message overhead, and is relatively easy to implement and debug.
What is Calvin's core idea for distributed transactions?	Choose a deterministic global transaction order before execution so replicas can execute consistently with less runtime coordination.
How does Calvin reduce coordination during execution?	Replicas agree on transaction order and boundaries in advance, then execute against the same ordered input without renegotiating conflicts.
What role does the Calvin sequencer play?	It batches incoming transactions into a deterministic global order.
What is a limitation of Calvin's deterministic approach?	It does not naturally support transactions whose read and write sets depend on additional ad hoc reads during execution.
Why can Calvin recover from node failures without aborting every transaction?	Replicas execute the same ordered transactions and can reconstruct state from the shared deterministic input.
What is Spanner's basic approach to distributed transactions?	It uses two-phase commit across partitions, with each partition backed by a consensus group for fault-tolerant replication.
Why does Spanner run two-phase commit over Paxos groups instead of individual nodes?	A participant can still make progress even if some nodes inside a group fail, improving availability relative to plain 2PC over single nodes.
What is TrueTime used for in Spanner?	To assign timestamps with bounded uncertainty and preserve externally consistent transaction ordering.
Why can read-only operations in Spanner often avoid the full cost of read-write transactions?	Versioned data lets many reads execute against stable snapshots without taking write locks.
What is external consistency in Spanner?	If transaction T1 commits before T2 starts, then T1 gets a smaller timestamp than T2, matching real-time order.
Why are multipartition transactions more expensive in Spanner than single-shard ones?	They need both per-shard consensus and a cross-shard two-phase commit.
Why is repartitioning necessary in a distributed database?	As nodes join, leave, or load changes, data must be moved to keep capacity and traffic balanced.
Why is simple modulo-based partition mapping unstable when cluster size changes?	Changing the divisor remaps most keys, forcing large-scale data movement.
What problem does consistent hashing solve?	It minimizes key remapping when nodes join or leave, so only a fraction of data needs to move.
How does consistent hashing place responsibility for keys?	Nodes sit on a hash ring and each becomes responsible for the range between its predecessor and itself.
What two guarantees define snapshot isolation?	Each transaction reads from a consistent snapshot, and conflicting concurrent writes to the same item cannot both commit.
What anomaly does snapshot isolation prevent compared with read committed?	Read skew caused by seeing mutually inconsistent committed values from different points in time.
Why is snapshot isolation still not fully serializable?	It can allow write skew, where disjoint writes each preserve local checks but jointly violate a global invariant.
What is Percolator's main approach to distributed transactions?	It builds snapshot-isolated transactions on top of Bigtable using timestamps, locks, and a client-driven two-phase commit.
What happens in Percolator's prewrite phase?	The transaction acquires locks and checks for conflicting later writes or unreleased locks before making tentative changes durable.
Why does Percolator designate a primary lock?	It gives recovery logic a single authoritative place to inspect when deciding whether an incomplete transaction should commit or roll back.
What happens in Percolator's commit phase?	The client releases locks beginning with the primary lock and publishes the write records so the new versions become visible.
Why can snapshot isolation improve read performance?	Readers can use stable snapshots without taking read locks on current mutable state.
What is coordination avoidance trying to achieve?	Preserve correctness while reducing the amount of cross-node synchronization needed for transactions.
What does I-confluence mean?	Any two independently produced invariant-valid states can be merged into another invariant-valid state.
Why does I-confluence matter for coordination avoidance?	If valid divergent states can always merge into a valid state, many operations can proceed without coordination.
What is the main idea of RAMP transactions?	Provide atomic visibility of multipartition writes without forcing mutual exclusion between readers and writers.
What does read atomicity in RAMP guarantee?	A transaction sees either all writes from another transaction or none of them, preventing fractured reads.
How does RAMP avoid blocking readers and writers on one another?	It uses metadata about in-flight writes so readers can detect missing pieces and fetch them if necessary.
Why do distributed transactions remain hard even with advanced protocols like Calvin, Spanner, Percolator, and RAMP?	They must balance atomicity, fault tolerance, coordination cost, and performance across partitions that can fail independently.
What problem do consensus algorithms solve in distributed systems?	They let multiple processes agree on a single value despite crashes and message delays.
Why can't consensus be guaranteed in bounded time in a fully asynchronous system?	Because a process cannot distinguish a crash from an arbitrarily slow process or network.
How do practical consensus systems get liveness despite FLP impossibility?	They rely on failure detectors or timing assumptions to suspect failed processes and keep the protocol moving.
What are the three core properties of consensus?	Agreement, validity, and termination.
Why is consensus useful in replicated systems?	It lets processes move through state changes in a common order without disagreement about what clients should observe.
What is the difference between best-effort broadcast and reliable broadcast?	Best-effort relies only on the sender to deliver messages, while reliable broadcast ensures all correct processes receive the same messages even if the sender fails.
Why is naive reliable broadcast expensive?	Recipients may have to rebroadcast to everyone else, producing roughly N^2 message traffic.
What does atomic broadcast add beyond reliable broadcast?	It guarantees all correct processes deliver the same messages in the same order.
What two properties define atomic broadcast?	Atomicity of delivery and a shared total order of delivered messages.
What is the relationship between atomic broadcast and consensus?	They are equivalent in asynchronous crash-failure systems because agreeing on message order is a form of consensus.
What is virtual synchrony?	A group-communication model where messages are delivered in a consistent order within a changing group view.
Why do view changes matter in virtual synchrony?	A message sent in one view can be delivered only in that same view, so views act as barriers around broadcasts.
What is the main idea of Zookeeper Atomic Broadcast (ZAB)?	A single leader orders updates and broadcasts them to followers, using epochs to keep leadership unique.
What happens in ZAB's discovery phase?	A prospective leader learns followers' latest epochs and proposes a new epoch greater than any they know.
What is the purpose of ZAB's synchronization phase?	To establish the new leader and bring followers to a consistent history before new broadcasts begin.
What happens in ZAB's broadcast phase?	The leader proposes updates, waits for quorum acknowledgments, and then commits them.
Why is ZAB efficient once a stable leader exists?	The leader can sequence requests with only a couple of message rounds instead of running fresh consensus for every event.
What roles exist in Paxos?	Proposer, acceptor, and learner.
What is the purpose of proposal numbers in Paxos?	They create a total order among competing proposals and prevent older proposals from overriding newer rounds.
What happens in Paxos phase 1?	A proposer sends Prepare requests and tries to collect promises from a quorum of acceptors.
What promise does an acceptor make in Paxos phase 1?	It promises not to accept future proposals with numbers lower than the prepare request it has acknowledged.
Why do acceptors report previously accepted values during Paxos prepare?	So a later proposer can preserve safety by continuing with the highest-numbered value already accepted.
What happens in Paxos phase 2?	The proposer sends Accept requests for the chosen value and tries to get a quorum of acceptors to accept it.
When must a Paxos proposer use an old accepted value instead of its own?	When a prepare response reveals that some acceptor already accepted a higher-numbered prior value.
Why are quorums central to Paxos safety?	Any two majorities overlap, so some acceptor carries forward information about previously accepted values.
What does Paxos guarantee with any number of failures?	Safety: it will not decide conflicting values.
What does Paxos require to guarantee liveness with f failures?	At least 2f + 1 processes so a quorum of f + 1 can still be formed.
How can Paxos recover from a proposer failure after partial progress?	A later proposer can learn the previously accepted value from an overlapping quorum and complete it.
Why can competing proposers cause repeated restarts in Paxos?	They can keep issuing higher-numbered prepares and prevent each other from completing phase 2.
How is proposer livelock commonly mitigated in Paxos?	With random backoff so one proposer eventually gets enough uninterrupted time to win.
What is Multi-Paxos?	A Paxos optimization where a stable leader skips repeating phase 1 for every value and directly replicates multiple log entries.
Why is Multi-Paxos much faster than rerunning classic Paxos for every command?	It avoids the full leader-establishment round on most operations.
Why do some Multi-Paxos implementations use leases?	To let the leader serve reads safely without a quorum round, as long as followers promise not to accept another leader for the lease period.
What is the main safety caveat of leases in Multi-Paxos?	They depend on bounded clock drift, so badly skewed clocks can break linearizable reads.
What is Fast Paxos trying to optimize?	It reduces one message delay by allowing proposals to reach acceptors more directly in fast rounds.
What trade-off does Fast Paxos make for lower latency?	It needs larger quorums and becomes more vulnerable to collisions between concurrent proposers.
What is the main idea of EPaxos?	Commands do not need a single global leader; instead, replicas infer ordering from dependencies between interfering commands.
When can EPaxos use its fast path?	When a fast quorum agrees on the same dependency set for the command.
When does EPaxos fall back to the slow path?	When replicas report conflicting dependency information that must be reconciled before commit.
What counts as interfering commands in EPaxos?	Commands whose execution order affects the final result.
Why can EPaxos improve throughput over leader-centric consensus?	Independent nonconflicting commands can commit without funneling all work through one leader.
What is Flexible Paxos?	A Paxos variant that requires only phase-1 quorums to intersect with phase-2 quorums rather than requiring all quorums to be majorities.
What does Flexible Paxos trade off?	It can reduce the replication quorum size for lower latency, but may require a larger quorum during leader election.
Why is the quorum condition Q1 + Q2 > N sufficient in Flexible Paxos?	Because it guarantees any phase-1 quorum overlaps with any phase-2 quorum.
What is the key simplification in Raft compared with Paxos?	Raft makes leadership explicit and separates leader election, log replication, and safety more clearly.
What roles exist in Raft?	Follower, candidate, and leader.
Why can a Raft node vote for at most one candidate per term?	To prevent multiple leaders from winning the same term.
What is a term in Raft?	A numbered time interval during which at most one leader is valid.
How does Raft resolve stale terms?	A node that sees a higher term updates its own term and yields to the newer one.
What does a Raft leader replicate?	An ordered log of commands for the replicated state machine.
When is a Raft log entry committed?	When the leader knows it has been replicated to a majority.
Why are only up-to-date candidates allowed to become leader in Raft?	So committed entries cannot be lost by electing a leader with an incomplete log.
What happens if a Raft follower's log conflicts with the leader's log?	The leader finds the latest common prefix and overwrites the follower's uncommitted conflicting suffix.
What guarantee does Raft provide about committed entries?	Committed entries remain in the logs of future leaders and cannot be rolled back.
How does Raft reduce the chance of split votes?	By using randomized election timeouts so one candidate is likely to start earlier than the others.
What extra problem do Byzantine consensus algorithms solve?	They tolerate nodes that behave arbitrarily or maliciously rather than merely crashing.
Why do Byzantine consensus protocols usually need more messages than crash-fault protocols?	Replicas must cross-check one another's claims instead of trusting the leader or a single response path.
How many nodes does PBFT need to tolerate f Byzantine faults?	At least 3f + 1 replicas.
What is the main structure of PBFT?	A primary orders requests, backups validate them, and replicas move through pre-prepare, prepare, and commit phases.
Why does PBFT use checkpoints?	To let replicas prove stable state, discard old protocol metadata, and support safe recovery.
Why is PBFT much costlier than crash-fault consensus?	It requires dense replica-to-replica communication and extra validation at every step.
Why must database subsystems be designed holistically?	Node-local performance, cluster communication, storage design, and coordination protocols all affect one another.
What usually has the biggest impact on node-local database performance?	The storage engine and local read-write path.
What often has the biggest impact on database scalability?	The subsystem responsible for communication and coordination across the cluster.
Why is a fast storage engine not enough on its own?	If it cannot scale with data growth or is paired with slow distributed coordination, overall system performance will still suffer.
Why is a fast storage engine undermined by a slow atomic commit protocol?	Distributed commit can become the dominant bottleneck even when local storage operations are fast.
Why is scalability necessary for a useful storage engine?	A storage engine with degrading performance as data grows supports only a narrow range of workloads.
What broad idea did Part II emphasize about distributed databases?	Node-local and cluster-wide mechanisms are interconnected and must be evaluated together.
What is the role of failure detection in distributed systems?	To identify remote process failures accurately enough for higher-level algorithms to keep making progress.
What is the role of leader election in distributed systems?	To choose a temporary coordinator that can reduce synchronization overhead.
What is the role of dissemination in distributed systems?	To distribute information reliably across nodes through communication protocols.
What is the role of anti-entropy in distributed systems?	To detect and repair replica divergence after updates are missed or delivered inconsistently.
What is the role of distributed transactions in distributed systems?	To execute operations across multiple partitions atomically.
What is the role of consensus in distributed systems?	To let remote participants agree on values despite failures.
Why is understanding these distributed algorithms practically useful?	It helps you choose infrastructure software more intelligently and reason about likely failure modes and trade-offs.
What kinds of systems commonly rely on these algorithms?	Databases, message queues, schedulers, and other infrastructure software.
