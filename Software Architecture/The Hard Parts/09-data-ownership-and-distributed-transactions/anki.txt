What happens after distributed data is pulled apart?	The data must be stitched back together through clear ownership, transaction handling, and access patterns for data a service no longer owns.
What is the general rule for assigning table ownership to services?	The service that performs write operations to a table owns that table.
Why does joint table ownership complicate the write-owner rule?	Multiple services write to the same table, so no single owner is obvious without applying another ownership technique.
How should architects usually handle single table ownership first?	Assign the writing service as owner first, then analyze harder common-ownership and joint-ownership cases.
What is common table ownership in distributed data design?	Most or all services need to write to the same table.
Why is a shared database a poor default solution for common table ownership?	It reintroduces shared-data problems such as change control, connection starvation, scalability limits, and fault-tolerance risk.
How can common table ownership be resolved?	Create a dedicated service as the only owner of the common table, and have other services send it write requests or events.
What is joint table ownership in distributed data design?	A small number of services, usually within the same domain, perform write operations on the same table.
What are common techniques for resolving joint table ownership?	Table split, data domain, delegate, and service consolidation.
What is the table split technique for joint table ownership?	Split one table into multiple tables so each service owns the columns or rows it is responsible for.
How does table splitting change ownership?	It turns joint ownership of one table into single ownership of separate tables.
What is a major synchronization problem with table splitting?	Services must coordinate creates, updates, or deletes that affect both split tables.
What consistency trade-off appears when splitting a jointly owned table?	Favoring availability lets one service proceed while related data may lag; favoring consistency may fail the operation if the other service is unavailable.
How does synchronous communication affect table splitting?	It improves confirmation and consistency between split tables but adds latency and availability dependency.
How does asynchronous communication affect table splitting?	It improves responsiveness and decoupling but can leave split tables only eventually consistent.
What is the data domain technique for joint table ownership?	Place the jointly written table in a shared schema or database owned by the participating services as a broader data domain.
What is the main advantage of the data domain technique?	It preserves data consistency and performance while avoiding service-to-service dependency for the shared table.
What is a major drawback of the data domain technique?	Schema changes require coordination across multiple services, increasing testing scope and deployment risk.
Why should teams reevaluate service boundaries when choosing a shared data domain?	Shared data may indicate the services are not independent enough to justify separate deployment unless other operational needs justify the split.
What is the delegate technique for joint table ownership?	Assign one service as the sole table owner and have other services ask it to perform reads or writes on their behalf.
Why is primary domain priority often preferred for the delegate technique?	It keeps domain management responsibility with the service that best understands the entity and its business rules.
What is a major disadvantage of the delegate technique?	It creates service coupling and interservice communication for nonowner reads or writes.
When is the delegate technique best suited?	For write scenarios that do not require atomic cross-service transactions and can tolerate eventual consistency.
What is the service consolidation technique for joint table ownership?	Combine the services that jointly write a table into one coarser-grained service.
What is the main transactional advantage of service consolidation?	It preserves atomic transactions inside one service and its database boundary.
What are major drawbacks of service consolidation?	It creates coarser-grained scalability, less fault isolation, larger testing scope, and higher deployment risk.
What should architects validate after assigning table ownership?	Business workflows and their transaction requirements.
When can a service in a distributed architecture still use ACID transactions?	Within its own database boundary, if the database supports ACID properties.
Why do distributed transactions not provide business-request atomicity?	Each service commits its own part independently, so the overall business request cannot commit or roll back as one unit.
What does BASE mean for distributed transactions?	Basic availability, soft state, and eventual consistency.
What are the main eventual consistency patterns discussed for distributed transactions?	Background synchronization, orchestrated request-based, and event-based.
What is the background synchronization pattern?	An external process periodically checks data sources and updates them to make them consistent.
What is the main responsiveness advantage of background synchronization?	The user-facing request can finish quickly because consistency work happens later outside the request path.
What is the biggest bounded-context problem with background synchronization?	The synchronization process needs access to multiple services' data sources, coupling data ownership and breaking bounded contexts.
Why can background synchronization duplicate business logic?	The external process may need to know service-specific rules for how data should be updated, archived, or deleted.
When is background synchronization a reasonable eventual consistency pattern?	For closed or heterogeneous systems that do not communicate directly and can tolerate slow consistency.
What is the orchestrated request-based pattern?	An orchestrator coordinates all service calls needed to make distributed data consistent during the user-facing business request.
What is a benefit of a dedicated orchestration service?	It keeps orchestration responsibilities out of domain services and centralizes the distributed workflow.
What does the orchestrated request-based pattern usually favor?	Data consistency and atomic business-request coordination over responsiveness.
Why is error handling hard in the orchestrated request-based pattern?	Some services may commit before another service fails, forcing retries, compensating transactions, or manual repair.
What is a compensating transaction?	A follow-up operation that attempts to undo or reverse already committed work after part of a distributed transaction fails.
What is the event-based pattern for eventual consistency?	A service publishes an event or command message, and interested services asynchronously react to update their own data.
Why is the event-based pattern popular in distributed architectures?	It keeps services decoupled, responds quickly to users, and often reaches consistency quickly through parallel asynchronous processing.
Why are durable subscribers important in pub/sub eventual consistency?	They ensure subscribers receive messages even if they were unavailable when the event was published.
What role does a dead letter queue play in event-based consistency?	It stores messages that repeatedly fail processing so automated repair or human intervention can handle them.
What is the main trade-off of the event-based eventual consistency pattern?	It improves responsiveness and decoupling but still requires careful error handling for failed event processing.
Why should read-only data needs not determine table ownership by themselves?	Ownership is based on write responsibility; read-only services should access nonowned data through approved cross-context access patterns.
Why can a service connecting to multiple schemas undermine bounded contexts?	It couples that service to multiple data ownership boundaries and recreates shared-database coordination problems.
How can delegation avoid a nonowner writing a table it needs to influence?	The nonowner sends the necessary data to the owner, often in an existing event or request, and the owner performs the write.
